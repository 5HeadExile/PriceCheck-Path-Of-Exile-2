# План: Item Price-Check по хоткею (как Exiled-Exchange-2)

> Отдельная, **самостоятельная** фича. Никак не трогает механику пилонов
> (`Scanning/`, `PriceOverlayForm`). Источник правды по статусу — журнал
> `.claude/skills/poe2-pricecheck/SKILL.md`.

## Контекст

Добавляем оценку **вещей** по горячей клавише: навёл на предмет → хоткей →
всплывает окно с разобранными модами (галочки + диапазоны значений) и списком
**живых листингов трейда** (кто и по какой цене продаёт). За образец — открытый
[Exiled-Exchange-2](https://github.com/Kvan7/Exiled-Exchange-2) (форк Awakened
PoE Trade, **MIT**), берём его подход и сгенерированные данные с атрибуцией.

**Решения пользователя:**
- Данные модов → stat id: **переиспользуем data-файлы EE2** (MIT + атрибуция).
- Окно: **как в EE2**, отдельный независимый модуль; пилоны не затрагиваем.
- Авторизация: **как у автора** — без ручного POESESSID; по умолчанию гость +
  rate-limit, опционально вход через встроенный браузер (WebView2).
- Улучшения в приоритете: **лучший UX модов**, **poe.ninja + трейд вместе**,
  **псевдо-моды** (total resistance/life).

## Как это работает (поток данных)

1. Глобальный хоткей (по умолчанию `Ctrl+D`) — программа эмулирует `Ctrl+C`,
   игра кладёт текст предмета в буфер обмена.
2. Читаем буфер → `ItemTextParser` разбирает формат PoE2 (секции, разделённые
   `--------`): класс, редкость, имя/база, ilvl, качество, сокеты/руны,
   имплициты/эксплициты, коррапт.
3. `StatDatabase` (данные EE2) сопоставляет строки модов → trade **stat id** и
   извлекает числовые значения; добавляем **псевдо-моды** (сумма сопротивлений и т.п.).
4. Окно показывает предмет + чекбоксы модов с min/max + быстрые пресеты; рядом —
   **быстрая оценка poe.ninja** (для баз/униcourse/валюты).
5. `TradeQueryBuilder` собирает запрос → `TradeClient`: POST в официальный
   **trade2 API** → GET листингов → список (аккаунт, цена, whisper).

## Авторизация и лимиты (ключевое)

- **По умолчанию — без логина** (гостевой доступ): запросы с корректным
  `User-Agent`, **строгий `RateLimiter`** читает заголовки `X-Rate-Limit-*` /
  `Retry-After`, ставит запросы в очередь и делает backoff на `429`. Это
  воспроизводит текущий опыт: работает, но при частых проверках — временный
  тайм-аут.
- **Опционально — вход через WebView2** (аналог встроенного браузера EE2):
  пользователь логинится один раз в окне, мы забираем cookie `POESESSID` из
  WebView2 CookieManager и прикладываем к запросам → выше лимит. Без ручной
  вставки cookie.

## Структура (новый модуль `ItemCheck/`, изолирован)

```
src/PriceCheckPoe2/ItemCheck/
  Input/ClipboardItemReader.cs     # эмуляция Ctrl+C (SendInput) + чтение буфера
  Parsing/ParsedItem.cs            # модель предмета
  Parsing/ItemTextParser.cs        # парсер буфера PoE2
  Data/StatDatabase.cs             # загрузка данных EE2, матчинг мод→statId+value
  Data/PseudoRules.cs              # псевдо-моды (total res/life)
  Trade/TradeQueryBuilder.cs       # сборка JSON-запроса trade2
  Trade/TradeClient.cs             # POST search + GET fetch (батчи по 10)
  Trade/RateLimiter.cs             # очередь + backoff по заголовкам лимитов
  Auth/SessionProvider.cs          # WebView2-логин (опц.), хранение cookie
  Pricing/NinjaEstimate.cs         # быстрая оценка через существующий PoeNinjaClient
  UI/ItemCheckWindow.xaml(.cs)     # WPF-окно (моды + листинги), независимое
  Data/ee2/                        # вендоренные data-файлы EE2 (+ATTRIBUTION)
```

Точки соприкосновения с существующим кодом — **только аддитивные**:
`HotkeyManager` (новый хоткей с модификатором), `AppConfig` (новые поля),
пункт в трей-меню. `PylonScanner`/`PriceOverlayForm` не меняются.

## Что переиспользуем

- **Данные EE2**: `stats`/`items` (ndjson/json, маппинг текста модов с `#` →
  stat id, базы) — вендорим в `ItemCheck/Data/ee2/` с файлом ATTRIBUTION (MIT).
- **Существующий `PoeNinjaClient`** — для inline-оценки (улучшение №2).
- **Существующий `HotkeyManager`** — расширяем поддержкой модификаторов (Ctrl/Alt).

## Зависимости (новые)

- `Microsoft.Web.WebView2` — встроенный браузер для логина (опц.).
- WPF уже включён (`UseWPF`); `MahApps.Metro` уже в проекте — пригодится для окна.
- HTTP — `HttpClient` + `Newtonsoft.Json` (уже есть).

## Улучшения сверх EE2

1. **poe.ninja + трейд вместе** — мгновенная оценка рядом с живыми листингами.
2. **Псевдо-моды** — авто total resistance / total life для точного поиска.
3. **Лучший UX модов** — разумные дефолты (предвыбор важных модов), память
   последних фильтров, 1-клик пресеты (только база / с модами / weighted).

## Поэтапные вехи (серия I — item check)

- **I0** — разведка + этот план + вендоринг data-файлов EE2 (с атрибуцией).
- **I1** — `ClipboardItemReader` + `ItemTextParser` + тесты на сэмплах текста.
- **I2** — `StatDatabase` (данные EE2) + матчинг мод→statId + `PseudoRules`.
- **I3** — `TradeClient` + `RateLimiter` (гость) + `TradeQueryBuilder`.
- **I4** — WPF `ItemCheckWindow`: предмет + выбор модов + список листингов;
  хоткей `Ctrl+D`; трей-пункт.
- **I5** — inline poe.ninja, пресеты, память фильтров (улучшения).
- **I6** — опц. вход через WebView2 для повышенного лимита.
- **I7** — тесты, docs, сборка на Windows.

## Верификация

- **Unit (без сети):** `ItemTextParser` на сохранённых сэмплах буфера разных
  типов предметов; `StatDatabase` матчинг; `TradeQueryBuilder` (ожидаемый JSON);
  `PseudoRules`. `dotnet test`.
- **Интеграция:** реальный `search`+`fetch` к trade2 (гость), лог лимитов;
  проверка backoff на `429`.
- **E2E (Windows):** в игре `Ctrl+D` на предмете → окно с модами и листингами;
  проверить независимость от оверлея пилонов.

## Риски

- **Формат данных/stat id меняется между лигами** — вендорим EE2 + предусмотреть
  обновление; держать версию данных в конфиге.
- **Cloudflare/блокировки гостя** — фолбэк на WebView2-логин.
- **Rate limit / возможный бан** — корректный `RateLimiter` обязателен (читать
  заголовки, не долбить, очередь).
- **Ctrl+C не везде копирует** (некоторые окна игры) — обрабатывать пустой буфер.
- **Лицензия** — MIT: добавить ATTRIBUTION к EE2 и Awakened PoE Trade.
