## Цель и статус

**Цель.** Десктоп-приложение (оверлей) для PoE2 с двумя независимыми фичами:
1. **Оценка пилонов** (механика «Runes of Aldur») — OCR области + poe.ninja + EV.
2. **Price-check вещей по хоткею** (как Exiled-Exchange-2) — Ctrl+C предмета →
   парсинг модов → официальный trade2 API → живые листинги. План:
   [`docs/PLAN-ITEMCHECK.md`](../../../docs/PLAN-ITEMCHECK.md). Отдельный модуль
   `ItemCheck/`, пилоны не затрагивает.

**За основу взят** открытый проект
[PoeAncientsPriceHelper](https://github.com/pedro-quiterio/PoeAncientsPriceHelper)
(C#/.NET 8 оверлей, Tesseract OCR + poe.ninja). Дорабатываем и улучшаем.

**Текущий статус:** собрано/протестировано на Windows (.NET 8.0.422), 21/21 тест,
exe в `dist\`. Endpoint poe.ninja PoE2 **подтверждён**, цены реально считаются,
имена матчатся по живому прайсу, оверлей рисует цену напротив каждой награды.
Добавлен авто-режим (`RegionMonitor`): оверлей сам показывается/скрывается по
открытию пилона.

**Фича 2 (item price-check):** I0–I5 готовы, работает end-to-end. Данные EE2
завендорены (MIT). Модуль `ItemCheck/`: `ClipboardItemReader`, `ItemTextParser`,
`StatDatabase`+`PseudoRules` (мод→trade stat id), `TradeQueryBuilder`+`RateLimiter`+
`TradeClient` (гость), WPF `ItemCheckWindow` (STA-поток), `ItemCheckService`
(оркестратор) + inline poe.ninja. Хоткей **Ctrl+D**, пункт в трее. **45/45 тестов**,
exe в `dist\`. Пилоны не тронуты. Осталось: живой тест и I6 (WebView2-логин, опц.).

**Текущая лига PoE2:** «Return of the Ancients» (0.5.0), механика — «Runes of
Aldur» (это и есть пилоны). На poe.ninja slug лиги — `runesofaldur`, имя для
API — «Runes of Aldur» (дефолт в конфиге).

**План целиком:** [`docs/PLAN.md`](../../../docs/PLAN.md) (утверждён). Журнал —
источник правды по фактическому состоянию и отклонениям от плана.

## Решения (и почему)

- **Стек C#/.NET 8 (WPF + WinForms)** — максимальное переиспользование архитектуры
  референса, быстрый старт. Минус: только Windows (пользователь согласен).
- **Источник цен — poe.ninja (раздел PoE2).** Заложен интерфейс `IPriceSource`,
  чтобы при необходимости заменить на офиц. Trade API PoE2. Точный endpoint PoE2
  ещё не подтверждён — уточнить инспекцией сетевых запросов poe.ninja/poe2/economy.
- **Распознавание — OCR-оверлей** с разовой калибровкой области экрана.
- **Новая фича: запуск в трей + игровое меню по хоткею.** Приложение стартует
  свёрнутым в трей; меню открывается только глобальным хоткеем прямо в игре —
  экран затемняется полупрозрачным слоем, по центру всплывает окно меню; повтор
  хоткея/Esc закрывает. Требует windowed-fullscreen в игре.
- **Зависимости как в референсе:** Tesseract 5.2.0 (+English 4.0.0), SharpHook
  7.1.2, MahApps.Metro 2.4.10, Newtonsoft.Json 13.0.3.
- **Окно настроек — WinForms, не WPF/MahApps** (отклонение от плана). Причина:
  единый UI-стек с треем/оверлеем и надёжная сборка непроверенного кода; XAML/
  MahApps-полировку можно вернуть позже. Зависимость MahApps пока оставлена.
- **Endpoint цен ПОДТВЕРЖДЁН** (DevTools на Runes of Aldur, см. docs/ENDPOINT.md):
  `…/poe2/api/economy/exchange/current/overview?league={League}&type=Currency`.
  Ответ `{ core, lines, items }`: цена в `lines[].primaryValue` (в divine),
  имя — в `items[]` по `id`; перевод в exalted = `primaryValue*core.rates.exalted`.
- **Имена наград матчим по живому прайс-листу** poe.ninja (`RewardParser.FromNames`
  из ключей прайса), а не по статичному файлу — имена всегда совпадают с источником.
  `reward-aliases.json` остался лишь как fallback при недоступности цен.
- **Цену рисуем напротив каждой строки** через bounding-box'ы OCR
  (`OcrEngine` отдаёт `OcrLine{Text,Bounds}`, координаты → экранные в `PylonScanner`).
- **Авто-режим (фоновый `RegionMonitor`)** вместо ручного «Оценить». Идея из
  референса (`ScanEngine`/`ListDetector`): фоновый цикл, яркость правой части
  области с гистерезисом (2 светлых кадра — открыть, 3 тёмных — закрыть, мёртвая
  зона 80..100) определяет, открыта ли панель. Открыта → скан и показ цен;
  закрыта → скрытие. **Улучшение над референсом:** OCR запускается только при
  открытии/смене содержимого (детект по «сигнатуре» области), а не каждый тик —
  меньше нагрузки. Калибровка области сразу подхватывается циклом (авто-скан).
  В меню «Оверлей: пауза/возобновить» (RegionMonitor.Paused) и «Пересканировать».
- **Item price-check — отдельный модуль `ItemCheck/`**, полностью изолирован от
  пилонов (Scanning/PriceOverlayForm/RegionMonitor не трогаем; точки контакта —
  только аддитивные: HotkeyManager/AppConfig/трей, и то на I4).
- **Данные модов берём из EE2** (`stats.ndjson`/`items.ndjson`, MIT) — вендорим с
  пином коммита и ATTRIBUTION, не парсим игру сами. Схема: stat — `{ref, matchers,
  trade.ids.{explicit,rune,...}, id}`; item — `{name, refName, namespace, ...}`.
- **Парсер буфера = секции по `--------`**; моды разбираем только у снаряжения
  (Magic/Rare/Unique), у валюты/гемов строки секций — описание, не моды; вид
  аффикса — по суффиксу строки `(implicit|rune|enchant|...)`. Защита: строки
  «Ключ:» (Requirements/Level/…) не считаются модами.
- **Окно price-check — WPF в коде (без XAML) на своём STA-потоке** с собственным
  `Dispatcher.Run()`. Почему: надёжнее, чем мешать WPF-окно с циклом WinForms
  (Application.Run), и не нужен App.xaml/Application.Current. Окно изолировано,
  закрытие → `InvokeShutdown()`. Trade/ninja обновляют UI через `Dispatcher.Invoke`.
- **Конфликт имён WPF vs WinForms/Drawing.** В проекте глобально импортированы
  `System.Windows.Forms` и `System.Drawing`, поэтому `Button/CheckBox/Color/Brush`
  неоднозначны в WPF-файле. *Решение:* алиасы пространств `Sw=System.Windows`,
  `Swc=…Controls`, `Swm=…Media` и обращение через них. *Впредь:* во всех WPF-файлах
  проекта использовать эти алиасы.
- **Trade2 — гостевой режим + `RateLimiter`** (читает X-Rate-Limit-*, backoff на
  429). Логин WebView2 (I6) отложен — гость работает, при частых проверках лимит.
- **Inline poe.ninja** переиспользует `PriceCache`/`PoeNinjaClient` фичи пилонов
  (общий код ценообразования), но фичи остаются раздельными.

## Инструменты по моментам

| Задача | Инструмент / подход |
|---|---|
| Глобальные хоткеи (ловятся в игре) | `SharpHook` (low-level хук) |
| Трей-иконка, запуск без окна | WinForms `NotifyIcon` + `ApplicationContext` |
| Затемнение экрана + меню | Полупрозрачные WinForms-формы (TopMost, borderless) |
| Захват области экрана | GDI `BitBlt` / `Graphics.CopyFromScreen` |
| OCR | `Tesseract`, предобработка: grayscale + threshold + upscale (тёмный фон) |
| Нечёткое сопоставление имён наград | Левенштейн поверх `Data/reward-aliases.json` |
| HTTP к poe.ninja | `HttpClient` + `Newtonsoft.Json` |
| Окно настроек | WinForms `Form` (отклонение от плана; см. «Решения») |
| DPI | свойство проекта `ApplicationHighDpiMode=PerMonitorV2` (не манифест) |
| Bitmap → Pix (Tesseract 5.x) | `Pix.LoadFromMemory(PNG-байты)` (нет `PixConverter`) |
| Открыта ли панель | `ListDetector` — яркость правой части области + гистерезис |
| Детект изменения контента | «сигнатура» области (сумма яркостей сетки) в `RegionMonitor` |
| Авто показ/скрытие + авто-скан | фоновый `RegionMonitor` (Task-цикл) |

## Решения по item price-check (фича 2)

- **Парсинг — из буфера (Ctrl+C), не OCR.** Текст предмета надёжнее OCR;
  это стандарт Awakened PoE Trade / EE2.
- **Источник — официальный trade2 API** (`pathofexile.com/api/trade2/search` +
  `/fetch`, `realm=poe2`), а не poe.ninja: нужны живые листинги «кто продаёт».
- **Авторизация:** EE2 работает **без ручного POESESSID** — гость с лимитом;
  для повышения лимита — встроенный браузер-логин (Electron BrowserWindow +
  прокси). Наш аналог — **WebView2** (опц.). При частых запросах → `429`
  (тайм-аут) — отсюда «таймаут» у пользователя. Обязателен `RateLimiter`
  (заголовки `X-Rate-Limit-*`, очередь, backoff).
- **Данные мод→statId:** переиспользуем сгенерированные файлы EE2 (MIT +
  атрибуция), вендорим в `ItemCheck/Data/ee2/`.
- **Окно — WPF**, отдельный модуль `ItemCheck/`, независим от пилонов.
- **Улучшения:** poe.ninja inline + псевдо-моды + лучший UX выбора модов.

## Лог ошибок (что → как решил → как не повторять)

- **Окружение сборки.** Контейнер Claude Code на Linux, `dotnet` отсутствует, а
  проект таргетит `net8.0-windows` (WPF/WinForms нельзя собрать на Linux).
  *Решение:* пишем исходники здесь, сборку/тесты (`dotnet build`/`dotnet test`)
  выполняет пользователь на Windows. *Впредь:* не обещать «собрано/проверено» из
  этого окружения; явно помечать Windows-only шаги как непроверенные локально.
- **poe.ninja PoE2 endpoint не подтверждается из песочницы.** Прямые запросы к
  `poe.ninja/poe2/api/...` дают 404 (сайт на Next.js, нужны браузерные заголовки/
  buildId-путь). *Решение:* endpoint вынесен в конфиг, парсинг устойчив к формам;
  финальную сверку делать в браузере (DevTools → Network на poe.ninja/poe2/economy).
  *Впредь:* не «подтверждать» внешний API по 404 из песочницы — помечать как
  требующий проверки и делать заменяемым.
- **Старый config.json «замораживал» каталог категорий.** Руны/эссенции и пр. не
  оценивались, т.к. сохранённый `config.json` держал `PriceOverviews=["Currency"]`
  и переопределял новый дефолт. *Решение:* `[JsonIgnore]` на встроенных каталогах
  (`PriceOverviews`) — берутся из кода, не из сохранённого конфига. *Впредь:*
  встроенные справочники не персистить вместе с пользовательскими настройками.
- **Плашки оверлея отдавали в фиолетовый.** Сначала думал — полупрозрачный фон;
  сделал непрозрачным (alpha=255), но фиолет остался. Настоящая причина:
  **сглаживание (AntiAlias) краёв скруглённых плашек/рамок и ClearType-текст
  смешивались с `TransparencyKey=Magenta`**, давая фиолетовую кайму по контуру.
  *Решение:* сменил ключ прозрачности на **чёрный** (кайма уходит в тёмное) и
  текст рисую серым AA, а не ClearType. *Впредь:* в click-through оверлее ключ
  прозрачности — нейтрально-тёмный, не яркий; и не ClearType на прозрачном окне.
- **Тесты на internal-методы.** `RewardParser.Levenshtein` и
  `PylonScanner.SplitCount` — `internal`; тесты в другой сборке. *Решение:*
  `InternalsVisibleTo("PriceCheckPoe2.Tests")` в `Properties/AssemblyInfo.cs`.
  *Впредь:* при тестировании internal-логики сразу добавлять этот атрибут.
- **Первая сборка на Windows — 3 класса ошибок (исправлены).**
  1) `SharpHook` 7.x: enum `KeyCode` лежит в `SharpHook.Data`, а не
     `SharpHook.Native`. *Впредь:* для SharpHook 6+ использовать `SharpHook.Data`.
  2) **WinForms-проект даёт урезанный набор implicit usings** — без `System.IO`
     и `System.Net.Http` (видно в `obj/.../GlobalUsings.g.cs`). *Решение:* вернул
     их через `<Using Include=... />` в csproj. *Впредь:* в WinForms/WPF не
     рассчитывать на эти namespace по умолчанию.
  3) **Tesseract 5.x API:** нет `PixConverter` (→ `Pix.LoadFromMemory(PNG)`);
     `ImageFormat` конфликтует с `System.Drawing.Imaging.ImageFormat` (→ алиас
     using); конструктор `TesseractEngine(path,"eng")` без `EngineMode`.
  Также DPI из манифеста → `ApplicationHighDpiMode` (варнинг WFAC010).
- **Первый живой тест: всё оценилось в 0.0 ex.** Скрин окна «Runeshape
  Combinations» показал две причины: (1) дефолтный endpoint был неверный
  (`currencyexchange/overview` с `leagueName`/`overviewName` → 404, прайс пустой);
  (2) `reward-aliases.json` содержал имена PoE1 (Exalted/Divine/Chaos), а в окне —
  PoE2-орбы (Jeweller's/Gemcutter's/Glassblower's…), совпадений ноль. *Решение:*
  подтвердил endpoint через Playwright DevTools, переписал `PoeNinjaClient` под
  реальную схему, имена матчим по живому прайсу. *Впредь:* источник цен и словарь
  имён проверять на реальных данных игры, не на догадках.

## Лог ошибок (фича 2)

- **Ctrl+D не срабатывал.** Отслеживание Ctrl отдельными KeyPressed/KeyReleased
  оказалось ненадёжным. *Решение:* брать состояние модификатора из маски события
  `e.RawEvent.Mask` (в SharpHook 7.x тип — **`EventMask`**, не `ModifierMask`;
  флаги `LeftCtrl/RightCtrl`). *Впредь:* модификаторы хоткеев — через `EventMask`
  события, не через ручной трекинг. Плюс добавлены трей-подсказки и
  `itemcheck_log.txt` (буфер пуст / не распознан / ошибка) для диагностики.
